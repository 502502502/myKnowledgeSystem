##### 1、软件的定义

+ 一系列指令集合和相应的数据结构和文档
+ 发挥计算机性能的工具
+ 传递信息的一种工具
+ 信息的处理手段



##### 2、软件的特征

+ 抽象的逻辑元素
+ 可变
+ 可重用
+ 可定制
+ 可测试
+ 可移植
+ 创新性高



##### 3、软件危机产生的原因

+ 需求存在二义性
+ 软件复杂
+ 人员素质不高
+ 缺乏有效的软件开发方法和工具
+ 没有统一、规范的方法论
+ 无法准确估计经费和进度
+ 忽视测试阶段工作



##### 4、软件过程 

+ 开发和维护软件的一系列活动
+ 软件工程活动和软件管理活动



##### 5、软件过程能力 

遵循软件过程**能够实现预期结果的程度**



##### 6、软件过程性能

遵循软件过程得到的**实际结果**



##### 7、 CMM

软件过程的成熟度评估和改进模型



##### 8、成熟度级别及其主要特征

+   **初始级**   ：没有什么过程是经过定义的
+   **可重复级** ：已建立基本的项目管理过程
+  **已定义级**：管理活动和工程活动两方面的软件过程均已文档化、标准化
+ **已管理级** ：已采集有关软件过程和产品质量的度量
+ **优化级**    ：利用过程和新思想、新技术的定量反馈信息，持续改进过程




##### 9、软件过程的可视性

- 等级1―――无可视性
- 等级2――― 里程碑有管理可视性
- 等级3―――内部结构可视
- 等级4―――软件过程有度量
- 等级5―――对过程不断改进



##### 10、过程能力和性能预测与成熟度之间的关系

- 随着成熟度增长，实际结果相对预定目标结果的偏差范围减小
- 随着成熟度增加，预定目标结果得到改善



##### 11、KPA定义

关键过程区域，每个关键过程区域识别出一串相关活动，当这些活动全部完成时，能达到一组目标



##### 12、KPA结构

- 目标
- 实践
- 能力级别



- 共同特点
  - 执行**约定**
  - 执行**能力**：是企业实施KPA的前提条件；
  - 执行**活动**：描述了执行KPA所需求的必要行动、任务和步骤；**唯一一项与项目执行相关的属性**。
  - 度量分析
  - 验证实施
  
  

##### 13、CMM每个成熟度等级的KPA及其子目标

+ 初始级

  +  KPA：无

    子目标：没有明确定义的过程

+ 可重复级

  + KPA：**要素管理**

    子目标：配置管理、项目跟踪和监管、质量保证

  + KPA：**项目计划**
    子目标：计划制定、计划监控和调整、软件子计划和迭代计划、风险管理

+ 已定义级
  + KPA：**工程过程**
    子目标：工程过程的定义、工程过程的实施、组织培训、工程过程的改进
  + KPA：**产品集成**
    子目标：集成规划、接口定义和协调、集成测试、集成的软件建设

+ 已管理级
  + KPA：**量化管理**
    子目标：过程度量和分析、过程控制和调整、定量的软件过程改进
  + KPA：**过程改进**
    子目标：定量的过程改进、持续的过程改进

+ 优化级
  + KPA：过程创新
    子目标：新的技术和方法、新的工具和技术评估、改进的工作环境、定量的过程改进



##### 14、项目的定义

利用***有限的资源***，在***有限的时间***内为***特定客户***完成***特定目标***的***一次性***工作。



##### 15、项目特征

##### 16、项目管理的定义

##### 20、需求的定义

- 达到目标所需的条件或权能

##### 21、需求分析定义

与用户沟通，理解需求，提出**完整**、**正确**、**一致**、**清晰**的要求



##### 22、需求分析的过程

+ **准备阶段**：
+ **采集、澄清需求**：建立**系统目标和范围**、采集**功能**需求、采集**外部接口信息**、采集**环境**需求、采集**性能**需求、采集**标准需求**、采集用户**特殊需求**、**准备和评估原型**；
+ **分析需求**：设计**过程模型**、设计**逻辑数据模型**、建立**数据字典**；
+ 准备SRS
+ 评审SRS
+ 客户认可并签署SRS

+ 一致性

+ 在出现需求变更申请时，必须执行 **需求变更管理过程**。在进行需求跟踪管理时，**跟踪矩阵** 是常用的管理工具。

##### 28、软件开发每个子过程的参加者、五要素

+ 概要设计
  + 参加者：设计团队、评审团队、客户
  + 输入准则：**需求规格文档经过评审并授权**
  + 输入：需求规格文档
  + 输出准则：**概要设计文档经过评审和授权**
  + 输出：概要设计文档、项目标准、概要设计评审记录
  + 度量：概要设计工作量、概要设计缺陷、评审工作量、返工工作量
+ 详细设计
  + 参加者：设计团队
  + 输入准则：**概要设计文档经过评审和授权**
  + 输入：概要设计文档
  + 输出准则：**详细设计文档和单元测试计划已经经过评审和授权**
  + 输出：详细设计文档和单元测试计划
  + 度量：详细设计工作量、设计缺陷、单元测试缺陷、程序框架缺陷、评审和返工工作量
+ 编码和单元测试
  + 参加者：项目组成员 、项目经理
  + 输入准则：**详细设计文档经过评审并授权**
  + 输入：详细设计文档、项目标准、程序规格、单元测试计划、程序框架、可利用开发的实用程序工具及文档
  + 输出准则：**成功执行所有单元测试计划中的测试用例**
  + 输出：源代码、可执行代码、测试数据
  + 度量：编码和单元测试的工作量、代码评审缺陷、独立单元测试缺陷、评审和返工工作量
+ 集成测试
  + 参加者：集成测试团队
  + 输入准则：**概要设计文档经过评审和授权**
  + 输入：概要设计文档和程序
  + 输出准则：**成功执行所有集成测试计划中的测试用例**
  + 输出：源代码、可执行代码、测试数据
  + 度量：工作量、缺陷
+ 系统测试
  + 参加者：系统测试团队
  + 输入准则：**需求规格和概要设计文档经过评审和授权**
  + 输入：需求规格和概要设计文档
  + 输出准则：**成功执行所有系统测试计划中的测试用例**
  + 输出：源代码、可执行代码、测试数据
  + 度量：工作量、缺陷
+ 验收测试和安装
  + 参加者：安装团队、客户、项目经理
  + 输入准则：**成功的完成系统测试**
  + 输入：测试后的软件和验收测试文档
  + 输出准则：**客户签署验收单**
  + 输出：安装后的软件
  + 度量：工作量和缺陷
+ 系统维护
  + 参加者：安装团队、维护团队
  + 输入准则：**在运行的软件**
  + 输入：安装后的软件、用户文档和软件故障报告
  + 输出准则：**合同中指定的维护支持阶段终止及客户签署整个项目**
  + 输出：客户签署的文档、已经安装的软件
  + 度量：工作量、缺陷

##### 33、软件度量的含义、作用

- 软件度量的作用

  - 能够帮助SEPG分析和改进软件过程

  

##### 34、过程数据库定义、构成

- 定义：PDB 存储和管理企业各类业务流程信息的数据库系统，实现对业务流程的全面监控、分析和优化

- 构成：

  - 流程定义：包含了业务流程中所有的活动、步骤和操作
- 流程实例：具体业务流程的运行实例，比如当前状态和步骤
  - 流程数据：涉及的所有数据和文档
  - 流程日志：记录业务流程的活动、步骤和操作历史
  - 流程分析工具：包括报表、查询和可视化工具
  
  

##### 35、 过程能力基线的定义、数据及含义（应掌握PCB数据项的计算方法）

- 定义：已经开发完成的软件项目的过程度量结果和分析数据
- 数据：
  - 阶段划分和对应的过程度量指标
  - 每个阶段的过程改进措施和实践方式
  - 过程度量结果和分析数据
    - 工作量 = 总人天数 * 人均每日开发时间
    - 缺陷密度 = 缺陷总数 / 代码行数
    - 变更请求率 = 变更请求总数 / 代码行数
    - 缺陷修复时间 = 所有缺陷修复时间之和 / 缺陷总数

##### 45、风险的含义

**可能发生的事件或者条件**，对项目产生有害的或者负面的影响。



##### 46、风险管理的含义、内容、目标和特点

- 含义：使**意外事件**而导致项目失败的**概率降到最小**。

- 特点

  - 付出额外的成本
- 价值不容易度量
  
  

##### 47、风险管理主要包含哪两类活动

- 风险评估
- 风险控制



##### 48、  如何进行风险评估

- **风险识别**
  - 风险检查表
  - 会议
  - 评审
  - 调查
- **等级划分**
  - 根据风险暴露度划分（RE）
  - RE(r)=Prob (r) *Loss (r)



##### 49、确定风险级别的步骤

+ 将风险**发生的可能性**划分级别。
+  对**项目的影响**评估级别
+  基于风险的可能性及其带来的后果划分级别

##### 55、 配置管理的概念

对变更进行系统地控制



##### 56、 配置管理的功能

- 给出程序的状态、最新版本；
- 并发处理变更，取消/显示一个程序变更、需求变更，提供可追踪性，防止未授权的变更；
-  收集当前系统的所有源代码、文档和其他信息。



##### 57、 典型的配置项有哪些

+ 软件代码：软件代码、脚本文件、配置文件等。
+ 文档：技术文档、用户手册、培训资料等。
+ 测试数据：测试计划、测试用例、测试报告等。
+ 工具和库：开发工具、编译器、代码库等。
+ 项目文档：项目计划、变更请求、问题记录等。
+ 安全配置：防火墙策略、访问控制列表、证书等。
+ 环境配置：开发环境、测试环境、生产环境等。



##### 58、如何执行配置管理过程中的变更申请

- 在项目的整个生命期内维护该项目。

##### 70、常见缺陷分析的方法

+ 5W1H法：通过回答 who、what、when、where、why 和 how 六个问题，找出缺陷发生的原因和影响。

+ 鱼骨图法（也称为因果分析图或石川图）：绘制一条“鱼骨”，将缺陷作为鱼骨头，然后使用分支图形式来探讨造成缺陷的原因，最终找到影响缺陷产生的根本原因。

+ 故障模式与效应分析（FMEA）：通过对系统中可能出现的故障进行分析，找出缺陷的根本原因以及可能存在的风险和潜在的影响。

+ KJ法：通过收集分析团队成员的想法和意见，将具有相似或相关特征的意见加以归纳和整理，然后找出原因和解决方法。

+ 标准化检查清单法：使用标准化清单，检查系统或产品中存在的缺陷，并记录在清单上，然后分析每个缺陷的成因和影响，制定解决方案。 
+ Pareto 图法：通过统计每种缺陷出现的频率，找到缺陷出现的主要原因

 





### 计算分析



##### 1、功能点法估计软件规模的步骤（**能够根据案例数据计算估计软件规模**）

+ 计算未调整的功能点UFP（unadjusted function points），通过检查和分析软件需求，确定所需的输入、输出、查询、数据文件和界面等5类系统元素的数量和复杂度，然后根据这些数量和复杂度计算各条目的功能点和整个软件的功能点。
+ 计算技术复杂度因子TCF（Technical Complexity Factor），对14个因子的评估，并将各评估值相加，即可获得技术复杂度因子TCF∈[0,70]
+ 计算功能点：FP = UFP * (0.65+0.01*TCF)



##### 2、自底向上的工作量估计方法步骤（**能够根据案例数据计算估计工作量**）

任务分解

每个程序单元的复杂度定义

估计每个单元的编码工作量

各阶段的工作量

计算整个程序的编码工作量 

导出整体项目的工作量 

+ 生产率=规模(FP)/工作量(人月)



##### 3、自顶向下的工作量估计方法步骤（**能够根据案例数据计算估计工作量**）

- 获得按照功能点计算的软件规模估计
- 使用类似项目的PCB数据或PDB数据中生产率数据确定项目的生产率级别
- 从生产率和规模估计获得整体的工作量估计
- 使用从PCB获得的工作量分布数据估计各阶段的工作量
- 修正估计，将项目特定因素考虑进去



##### 76、掌握用COCOMO初级、中级模型估计工作量和开发时间（**模型参数和查找表会给出**，**不需要记背）**

+ 估计未调整的功能点数
+ 计算技术复杂度因子，调整后得出调整后的功能点数
+ 将功能点数折算成千行代码行数
+ 分析案例描述，将项目进行分类（有机类、半分离类、嵌入式类）
+ 查找对应类别的模型系数，将千行代码行数和模型系数a,b,c,d代入基本COCOMO模型即可进行工作量估计
+ 如果使用中级COCOMO模型，需要在估计工作量调整因子EAF的基础上，将千行代码行数，模型系数a,b,c,d以及EAF代入中级COCOMO模型即可进行工作量估计
